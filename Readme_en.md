<a id="user-content-redmineundevgit" class="anchor" href="#redmineundevgit" aria-hidden="true"><span class="octicon octicon-link"></span></a>RedmineUndevGit</h1>

<p><a href="https://travis-ci.org/Undev/redmine_undev_git"><img src="https://camo.githubusercontent.com/21b461964dfc04796bf25c9c6e0565bed09d5479/68747470733a2f2f7472617669732d63692e6f72672f556e6465762f7265646d696e655f756e6465765f6769742e706e67" alt="Build Status" data-canonical-src="https://travis-ci.org/Undev/redmine_undev_git.png" style="max-width:100%;"></a>

<h2>
<a id="user-content-description" class="anchor" href="#description" aria-hidden="true"><span class="octicon octicon-link"></span></a>Description</h2>

<p>The UndevGit plugin adds the UndevGit repository type to Redmine.
UndevGit has all standard functions of a Git repository, as well as other features:
possibility to work with remote repositories;
hooks for UndevGit repositories.</p>

<p>When accessing a remote or local repository, UndevGit clones it 
and then works with the created copy.</p>

<h2>
<a id="user-content-setup-and-configuration" class="anchor" href="#setup-and-configuration" aria-hidden="true"><span class="octicon octicon-link"></span></a>Setup and configuration</h2>

<h3>
<a id="user-content-setup" class="anchor" href="#setup" aria-hidden="true"><span class="octicon octicon-link"></span></a>Setup</h3>

<ol class="task-list">
<li>Copy the plugin directory into <strong>plugins</strong>
</li>
<li><strong>bundle exec rake redmine:plugins:migrate</strong></li>
<li>
<strong>bundle install</strong> in the Redmine root directory</li>
<li>Restart Redmine</li>
</ol>

<h3>
<a id="user-content-configuration" class="anchor" href="#configuration" aria-hidden="true"><span class="octicon octicon-link"></span></a>Configuration</h3>

<p>To enable UndevGit, go to the Redmine administrative menu &gt; <em>Repositories</em> tab &gt; <em>Enabled SCM</em> group.</p>

<p>The plugin settings allow you to specify the maximum number of branches to be displayed on 
the issue page in the <em>Associated revisions</em> section.
An empty field or 0 means no restrictions.
For details on hook configuration, see the corresponding section below.</p>

<p>In the Redmine settings, you can specify the directory 
for storing local repository copies by setting
a value for the <code>scm_repo_storage_dir</code> key in the <code>config/configuration.yml</code> file.
By default, the <code>repos</code> directory in the Redmine root directory is used, so
local repository copies are stored in <code>repos\[projectidentifier]\[repository_id]</code>.
When you remove a repository, the corresponding directory is deleted.</p>

<h2>
<a id="user-content-hooks" class="anchor" href="#hooks" aria-hidden="true"><span class="octicon octicon-link"></span></a>Hooks</h2>

<p>Hooks are used to flexibly configure how and when an issue status can be changed.
You can configure hooks to be executed for certain branches, a project or even a specific repository.
Thus, there are global hooks that are run for all repositories, 
project hooks that are run for all project repositories, and repository hooks.</p>

<p>If '*' is specified as a branch for the hook, it means that this hook is applied only once when a commit is added to the repository;
if branch names are specified, the hook is applied each time a commit is added to a branch (once per branch).</p>

<h3>
<a id="user-content-configuration-1" class="anchor" href="#configuration-1" aria-hidden="true"><span class="octicon octicon-link"></span></a>Configuration</h3>

<p>Hooks implement the Redmine feature of changing the issue status
using keywords. Therefore, the keywords specified
on the <em>Repository</em> tab of the Redmine settings (<em>Fixing keywords</em> attribute)
are not used. You should define a set of keywords for each hook.</p>

<p>Hooks are configured as follows. Global hooks are managed from the Redmine settings window
(<em>Global hooks</em> menu item, above <em>Plugins</em>).
Project and repository hooks are available on the <em>Hooks</em> tab of the project settings menu.</p>

<h3>
<a id="user-content-hook-priorities" class="anchor" href="#hook-priorities" aria-hidden="true"><span class="octicon octicon-link"></span></a>Hook priorities</h3>

<p>Hook priorities are determined as follows: the repository hooks have the highest priority,
then go the project hooks, while global hooks have the lowest priority.
There are also different priorities within each hook type, which can be changed if needed.
If there are several hooks applicable to a commit,
only the hook with the highest priority will be run.</p>

<p>Hooks that are applied to all branches have a lower priority
than hooks with branch names specified explicitly.</p>

<h3>
<a id="user-content-logging" class="anchor" href="#logging" aria-hidden="true"><span class="octicon octicon-link"></span></a>Logging</h3>

<p>All changes of an issue are recorded in its history log. If no changes have been made,
the log will have no entries.</p>

<h3>
<a id="user-content-examples" class="anchor" href="#examples" aria-hidden="true"><span class="octicon octicon-link"></span></a>Examples</h3>

<h4>
<a id="user-content-example-1" class="anchor" href="#example-1" aria-hidden="true"><span class="octicon octicon-link"></span></a>Example 1</h4>

<p>A commit is pushed to 2 branches (feature, develop) of a repository.
There are 2 configured hooks:
Hook1 for branch '*'
Hook2 for branch 'feature'
In this case, only Hook2 will be run.</p>

<h4>
<a id="user-content-example-2" class="anchor" href="#example-2" aria-hidden="true"><span class="octicon octicon-link"></span></a>Example 2</h4>

<p>There are several hooks:
Hook1, global, branch '*'
Hook2, global, branch 'master'
Hook3, for a project, branch 'develop,staging'
Hook4, for a repository, branch 'staging'
Hook5, for a repository, branch 'feature'</p>

<p>First push: add commit A to branch 'feature' — Hook5 is run.
Second push: merge 'feature' to 'staging' — Hook4 is run.
Third push: merge 'feature' to 'develop' — Hook3 is run.
Fourth push: merge 'feature' to 'master' — Hook2 is run.</p>

<p>Hook1 is used only if commit B is pushed to branch 'featureX'.</p>

<h2>
<a id="user-content-linking-commits-to-issues" class="anchor" href="#linking-commits-to-issues" aria-hidden="true"><span class="octicon octicon-link"></span></a>Linking commits to issues</h2>

<p>To link a commit to an issue, specify the keyword and the issue number starting with #,
for example: <code>refs #124</code>.
Keywords are set on the <em>Repository</em> tab of the Redmine settings
(<em>Referencing keywords</em> attribute).
If you set '*' as a keyword, specifying only the issue number with #
will be enough to link a commit to the issue.</p>

<h2>
<a id="user-content-moving-of-commits-rebase" class="anchor" href="#moving-of-commits-rebase" aria-hidden="true"><span class="octicon octicon-link"></span></a>Moving of commits (rebase)</h2>

<p>UndevGit enables you to determine which commits have be moved by using <code>git rebase</code>.
Old references to commits are not removed from the changeset list; instead, they are marked with a special icon
linking to a new commit reference. Similarly, new commits are marked with icons
linking to old commit references.
This icon is available when viewing the list of commits or a specific commit.
Using rebase does not cause new changes of issues; however, the links in the <em>Associated revisions</em> section are changed.
Using rebase also does not affect timelogs.</p>
